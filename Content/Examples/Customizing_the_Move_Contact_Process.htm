Customizing the Move Contact Process

The Move Contact process provides several opportunities for customization.

How Do I?

Secure access to Move or Copy action

On the Contact Detail view toolbar, the Move and Duplicate Contact buttons have their own secured actions assigned to them â€” ENTITIES/CONTACT/MOVE and ENTITIES/CONTACT/COPY, respectively.

  • Manage access to the buttons independent of each other by assigning their secured actions to the appropriate roles.
    For instance, you could limit access to Duplicate Contacts by removing  ENTITIES/CONTACT/COPY from the standard user role.

Refer to "What are Roles and Secured Actions?" in the Saleslogix Web Client Help for more information.

Override business rule logic for an entity

The Contact.MoveContact business rule has a post execute step for each entity type associated with contacts.

  • You can override the move behavior for an entity type by creating a replacement post execute step for it.
Remove an entity type from the Move Contact process
  • If you want to remove an entity such as tickets from the Move Contact Process, remove its corresponding post execute step in the Contact.MoveContact business rule.
Add an entity type to the Move Contact process

You can incorporate additional entity types including custom ones into the Move Contact process.

To add

  1. The Move Contact dialog gives the user choices about moving associated records such as whether they should move, and if so, to what contact. If you want to offer move options for the entity type you are adding to the process, modify MoveContact.aspx.
  2. In the code behind, add a section for the new entity type that loads the move options into a moveContactArguments object. For example, copy and modify the block for Literature Requests:

    //Literature Requests Options
    moveOption = new MoveItemOption
     {
         Entity = "LitRequest",
         OpenItemsOnly = true,
         ClosedItemsOnly = true,
         ReassignTo = openItemsReassignContact.Id,
         ReassignToClosed = closedItemsReassignContact.Id,
         MoveToAccount = false
     };
    arguments.Items.Add(moveOption);

  3. To the Contact.MoveContact business rule, add a post execute step containing the move logic for the new entity type. For example, here is the  post execution step for Literature Requests, MoveContactLitRequests:
