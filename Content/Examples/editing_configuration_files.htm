Editing Configuration Files

The web.config and appSettings.config files (associated with ASP.NET) act as containers for settings that describe how the ASP.NET-specific portions of the application should be handled.

The configuration files must contain only valid XML to avoid problems with your application.

 

There are multiple copies of the configuration files. Tell me more...

  • Model copies - Web.config and appSettings.config files can exist for each portal definition in the model.
  • Website copies - When the portal is deployed to a target, the configuration files are copied to the root of the website. For example, web.config and appSettings.config for the Saleslogix (Web Client) portal might be located under C:\Inetpub\wwwroot\SLXClient.

Choose the copy you want to edit. Tell me more...

  • Model copies - The advantage of editing a configuration file in the Portal Explorer is that your changes are propagated to the Web server each time you deploy, for every deployment target.
  • Website copies - The advantage of editing the Web server copy of a configuration file instead is that changes made for debugging purposes are less likely to be reproduced in production deployments. The disadvantage is that each time the portal is deployed, this copy of the configuration file is overwritten.

To edit a model copy

  1. Open the portal for editing. See "Editing a Portal" in the Application Architect online Help.
  2. Expand the  Support Files folder, right-click the configuration file, and then click Open.
  3. Make your changes.
  4. Click Save.
  5. Redeploy the portal. See "Redeploying Portals" in the Application Architect online Help.

To edit a website copy

  1. Browse to the base directory of the portal on the Web server, for example, C:\Inetpub\wwwroot\.
  2. Double-click on the folder for the portal, for example, SLXClient.
  3. Open the configuration file with the editor of your choice.
  4. Make your changes.
  5. Save the file.

    When the web.config file is saved, ASP.NET restarts itself, which closes the portal and prompts users to log back in.

  6. Each time the portal is deployed, this copy of the configuration file is overwritten, so remember to port your changes back to the model.
