Understanding Process Orchestration

Process Orchestration functionality permits you to control the flow of work in Saleslogix. Process Orchestration workflows can be bound to any kind of Saleslogix entity.

Process Orchestration includes the following components:

  • Process Host portal
  • Entity Eventing Service
  • Integration with Web Service

sequential processes

The steps in a sequential process proceed in order from start to finish. An event, such as creation of an entity, will initiate one or more actions. For example, a sequential process could specify that on creation of a Lead entity, the "Lead Ownership" policy is executed. The policy details might be:

IF Lead.State == "AZ"

THEN Lead.Owner = "Bob"

ELSE Lead.Owner = "Bill"

Sequential processes usually run from start to finish without interruption.

Goal-driven processes

A goal-driven process contains multiple steps that the user can complete non-sequentially. Following are some characteristics of goal-driven workflows:

  • Steps are the smallest unit of work.
  • Steps can be grouped into a stage. Completion of all required steps within a stage completes the stage.
  • A workflow consists of a series of stages.
  • Steps can be completed outside of the currently "active" stage.
  • Stages can be grouped into scopes.
  • All stages within a scope must be finished before you can proceed to steps and stages in another scope. For example, you might use one scope for all pre-sales activities.
  • Goal driven processes wait for user input such as marking steps as complete or cancelled.

Types of Activities (steps) supported

You can use any Microsoft or Saleslogix activities to build your workflows. Saleslogix pre-defined activities include:

  • Schedule Activity: To Dos, Phone Calls

  • Goal driven steps (GoalActivity), stages (GoalDrivenActivity), and scopes (GoalDrivenScopeActivity)

 

Windows Policy Activities can be used to:

  • Assign Ownership (SecCodeID)
  • Set User (UserID)

Entity Eventing Service

  • Managed code entity eventing service integrates with data change events.
  • It can be configured to start workflows based on data changes. Qualifying database events can be pre-filtered with conditions.
  • New data change events are intercepted at the OleDB provider level.
  • The service can be consumed by other applications.
